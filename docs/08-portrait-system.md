# 08 — Procedural portrait system

Player faces should be generated by a deterministic, layered portrait system,
not authored as one-off full portraits. The goal is for every real player and
newgen to have a persistent, recognizable face while keeping art production,
legal risk, and save compatibility under control.

The first art-direction target is captured in
[`images/portrait-pixel-art-direction-v1.png`](images/portrait-pixel-art-direction-v1.png):
front-facing sports portraits, hard pixel clusters, restrained shading, and
national-color shirts. It is a visual reference rather than a production sprite
sheet; production portraits remain assembled deterministically from recipe parts.

## Recommendation

Use a small stylized pixel portrait generator at 96×96. Each portrait
is assembled from reusable layers in a fixed order: body/clothing, head, skin
shading, hair, facial hair, eyes, brows, nose, mouth, age marks, accessories,
and temporary overlays such as bandages.

This is the best fit for the game because it gives:

- **Determinism:** the same player keeps the same face across devices and save
  files.
- **Scalability:** a modest art kit can create thousands of distinct-looking
  players.
- **Newgen support:** future generated players need no new manual art pass.
- **Consistent style:** all portraits share the same visual language.
- **Low legal risk:** real players are represented as game avatars, not copied
  from photos or made photorealistic.

## Identity and seeding

Every player should store or derive a stable `portraitSeed`:

```text
portraitSeed = hash(playerId)
```

For imported real players, `playerId` should be the stable internal ID that is
mapped from the external data import. For generated players, it should be the
same deterministic ID produced by the newgen system. If a future content import
changes external source IDs, the portrait seed must remain stable through the
internal player identity mapping rather than being regenerated accidentally.

The seed chooses feature variants and small offsets, while player data can
constrain eligible pools:

- age influences grey hair, receding hair, wrinkles, and face softness;
- gender can influence hair, brow, facial-hair, and head-shape pools without
  making the result rigid or stereotyped;
- country can choose shirt or national accent colors, and softly weights skin
  tone, natural hair color, and eye shape (see "Regional appearance weighting"
  below), but must never mechanically determine any single facial feature;
- traits can influence expression, for example confident, tense, neutral, or
  cheerful;
- injury state can add temporary overlays such as a plaster, tape, or bandage.

## Regional appearance weighting

Skin tone, natural hair color, and eye shape are chosen by the same per-seed
deterministic rolls as every other feature, but the rolls are weighted by a
coarse regional tier derived from `country` (`APPEARANCE_REGION_BY_COUNTRY` in
`generate.ts`). Grey and salt-and-pepper hair remain age-driven. The weights
model the known competitive racketlon pool rather than general population
demographics.

Recipe version 2 keeps version 1's random stream, so this weighting change only
changes skin, natural hair color, and eyes; unrelated facial features do not
randomly reshuffle for existing player seeds.

Rules that keep this from becoming a stereotype generator instead of a
plausibility nudge:

- Every tier keeps a nonzero weight on every catalog option. Very rare outcomes
  remain possible, while the European tiers make the darkest skin palettes
  exceptional enough to match the observed sport-specific player pool.
- The tiers are rough game-design buckets, not demographic data. They exist to
  avoid the pool looking randomly uniform across very different countries, not
  to encode precise population statistics.
- A country missing from the map falls back to a uniform roll rather than
  guessing a tier.
- Reviewed public cues always win over generated choices. If a known player
  looks wrong, add a sparse manual cue instead of creating a one-player rule in
  the generator.
- If a whole country looks wrong, adjust its tier or the tier's three weight
  tables directly; keep the nonzero floor so unusual outcomes remain possible.

## Art kit target

Start with enough parts to make the generator useful without overcommitting:

| Layer | MVP count | Notes |
|---|---:|---|
| Head shapes | 8 | Include subtle silhouette differences. |
| Skin palettes | 8 | Palette swaps rather than separate sprites where possible. |
| Hair styles | 12 | Include short, long, tied-back, bald, and receding options. |
| Hair palettes | 8 | Include greying variants. |
| Eyes | 6 | Keep differences readable at 64×64. |
| Brows | 6 | Useful for expression and age. |
| Noses | 6 | Small but identity-forming. |
| Mouths | 6 | Neutral, smile, focused, tense variants. |
| Facial hair | 8 | Optional layer. |
| Glasses/accessories | 6 | Optional layer. |
| Age/detail marks | 8 | Wrinkles, freckles, moles, cheek lines. |
| Shirts | 8 | Club/country-neutral plus national-color palettes. |
| Injury overlays | 4 | Temporary status indicators. |

This is roughly 80 small assets, but because most are combinable and recolorable
it can cover the initial real-player pool and many seasons of newgens.

## Software architecture

Build portraits as a replaceable presentation module behind a small interface.
Game simulation code should never import art assets, renderer components, canvas
helpers, or concrete generator implementations. It should only expose stable
player identity and the limited public metadata that portrait generation is
allowed to consider.

Recommended package shape:

```text
@racketlon/portraits
  src/contracts.ts      # stable interfaces and serializable recipe types
  src/generate.ts       # default deterministic recipe generator
  src/renderers/        # swappable SVG/canvas/PNG-layer renderers
  assets/               # default art kit, versioned with the generator
```

The app should depend on an injected `PortraitProvider` rather than constructing
the default generator directly:

```ts
type PortraitInput = {
  playerId: string;
  portraitSeed: string;
  ageYears?: number;
  gender?: "m" | "f";
  country?: string;
  publicCues?: Partial<PortraitRecipe>;
};

type PortraitProvider = {
  recipeFor(input: PortraitInput): PortraitRecipe;
  assetUrlFor?(recipe: PortraitRecipe): string;
  render?(recipe: PortraitRecipe): PortraitRenderable;
};
```

Replacement rules:

- Treat `PortraitRecipe` as a versioned interchange format, not as the private
  state of one renderer.
- Keep rendering separate from recipe generation, so a later implementation can
  swap pixel PNG layers for SVG, canvas, server-generated spritesheets, or a
  licensed portrait pack.
- Keep saves and imported content independent of concrete asset filenames where
  possible. Store stable seeds, recipe versions, and optional semantic overrides;
  let the active provider resolve those into assets.
- Register the provider at the app boundary. Tests and debug tools should be able
  to inject a fake provider that returns simple placeholder portraits.
- Add adapter code for future generators instead of changing player/domain models
  every time the art approach changes.

## Generator rules

The generator should produce a `PortraitRecipe` rather than only a finished
image. A recipe is easier to debug, snapshot-test, migrate, and override:

```ts
type PortraitRecipe = {
  version: number;
  seed: string;
  head: string;
  skinPalette: string;
  hair?: string;
  hairPalette?: string;
  eyes: string;
  brows: string;
  nose: string;
  mouth: string;
  facialHair?: string;
  accessory?: string;
  ageMarks?: string[];
  shirt: string;
  shirtPalette: string;
  offsets: Record<string, { x: number; y: number }>;
};
```

Recommended rules:

- Keep variant selection deterministic from `(portraitSeed, recipe.version)`.
- Use eligibility filters first, then seeded random choice within the filtered
  list.
- Store manual overrides as partial recipes for special cases, not as separate
  one-off rendering logic.
- Treat temporary injury overlays as UI state layered on top of the stable
  recipe, so a player does not get a new identity after injury or recovery.
- Version recipes. A new art kit or algorithm can ship as `version: 2` while old
  saves keep their existing faces until a deliberate migration is chosen.

## Manual cue workflow

Reviewed real-player corrections live in the gender-specific sidecars
`packages/content/data/portrait-cues-men.json` and
`packages/content/data/portrait-cues-women.json`, keyed by the stable imported
`playerId`. Content loading validates and merges both maps, and the app passes a
matching sparse entry to the presentation adapter when it builds `PortraitInput`.
The generated `world-bundle.json` never owns these cues, so `build:world` can
replace ratings and roster data without overwriting them; the build instead
fails if a cue points to an id that disappeared from the roster, appears in both
files, or is stored in the wrong gender file.

Omitting a field preserves its deterministic generated choice. Setting
`accessory` or `facialHair` to `null` explicitly removes a generated optional
feature, which allows a review to say both “wears glasses” and “no glasses.”
The concrete version-1 values and an editable JSON example are documented in
`packages/content/src/import/README.md`.

## Real-player policy

Real players should not be photorealistic or intentionally copied from photos.
Use the generator to create game representations with broad cues only:

- approximate age band;
- known hair color or glasses when public and characteristic;
- broad facial-hair presence when useful;
- consistent shirt or country color accents.

The most recognizable 30–50 players can later receive hand-tuned partial recipe
overrides. Everyone else should use the same generator as newgens. The player
experience target is recognition inside the game world, not exact likeness.

## Online photo sources

Racketlon websites and tournament reports sometimes show player photos, but do
not treat online images as automatically usable game assets. A public web image
is still protected by copyright and may also carry photographer, athlete, event,
or platform rights. Do not scrape, hotlink, bundle, trace, or AI-transform those
photos into shipped portraits unless the project has an explicit license or
written permission for that exact use.

Allowed uses for public online faces during planning:

- manually record non-sensitive, broad cues for a small hand-tuned override, such
  as hair color, glasses, or facial-hair presence;
- use photos as private art direction references for a stylized avatar that does
  not copy the source image or aim for a close likeness;
- keep source URLs and license/permission notes in the content pipeline if any
  public photo materially informs a real-player override.

Disallowed uses unless separately licensed:

- downloading ranking-page or social-media headshots into the repo;
- hotlinking remote image URLs in the app;
- using face-recognition or image-generation pipelines to reproduce an athlete's
  likeness from web photos;
- claiming a player portrait is official or photo-accurate.

If FIR, a photographer, or a player later grants permission, store the license
metadata beside the override and still prefer the same stylized generator so the
portrait fits newgens and unlicensed real players.

## Implementation path

1. Define the `PortraitRecipe` schema and deterministic recipe builder in the UI
   layer or a shared presentation package.
2. Add the first hard-edged pixel renderer while retaining the original SVG
   renderer as a fallback adapter.
3. Render recipes to stacked image layers in the player list/profile screens.
4. Add snapshot tests proving a fixed player ID produces a stable recipe.
5. Add a small debug gallery that shows hundreds of seeded portraits at once to
   find bad combinations quickly.
6. Add partial manual overrides for famous players only after the generator is
   already working.

## Open decisions

- Final canvas size is now 96×96; the remaining question is how much of the
  renderer should eventually move from integer-aligned SVG geometry to authored
  PNG layer sheets.
- Asset format: pixel PNG layers are fastest for a pixel style; SVG layers are
  easier to recolor and scale but may look less like pixel art unless carefully
  constrained.
- Storage: decide whether saves store the full recipe or only `portraitSeed` and
  `recipe.version`. Full recipe storage preserves faces better across generator
  changes; seed-only storage is smaller but makes migrations riskier.
